# WCAG 3.0 방향 접근성 가이드 (프로젝트용)

## 언제 쓰는지

- 새 화면·컴포넌트·폼·미디어 플레이어를 만들거나 리팩터링할 때
- 키보드만·스크린 리더·고대비·텍스트 확대 시나리오를 점검할 때
- 법적·조달 요구로 WCAG 2.x와 함께 “차세대 지침”을 의식해야 할 때

## 전제 (반드시 알 것)

- **WCAG 3.0은 아직 작업 초안(WD)**이며 문구·요구·준수 모델이 바뀔 수 있습니다. 공식 원문은 [W3C WCAG 3.0](https://www.w3.org/TR/wcag-3.0/)을 따릅니다.
- **WCAG 2를 대체하지 않습니다.** 실무 준수 주장은 여전히 [WCAG 2.2](https://www.w3.org/TR/WCAG22/)가 기준인 경우가 많습니다. 본 스킬은 **WCAG 3.0의 구조와 방향**을 프로젝트 습관으로 옮기기 위한 것입니다.
- 한국어 개요 참고: [WCAG 3.0 (ko.htmlspecs.com)](https://ko.htmlspecs.com/wcag-3.0/)

## WCAG 3에서 쓰는 구조 (적용 습관)

| 유형 | 의미 | 프로젝트에서의 행동 |
|------|------|---------------------|
| **핵심 요구 (Core)** | 맞추어야 하는 기본 막대 | 릴리스 전 자동+수동 검증 대상 |
| **보충 요구 (Supplemental)** | 더 높은 품질·엄격함 | 점진 도입·디자인 시스템에 편입 |
| **선언 (Assertions)** | 조직이 문서화해 주장하는 프로세스 | 팀 정책·체크리스트·감사 기록으로 관리 |

준수 수준(초안): **Bronze / Silver / Gold** — Bronze는 “기능 수행 진술”별 **모든 핵심 요구** + 보충·선언 일부(비율은 명세 확정 시 정의)를 전제로 합니다.

## 기능 수행 진술을 염두에 둔 설계

요구사항은 “장애 목록”이 아니라 **사용 맥락**에 태그됩니다. 구현 시 다음을 빠뜨리지 않습니다.

- 시각 없이 / 제한된 시력 / 색 인지 없이
- 청각 없이 / 제한된 청력
- 제한된 조작·도달 범위
- 감각 자극 민감도, 제한된 주의·실행 기능·추론·기억·언어 처리
- 특정 콘텐츠 민감도, 비전형 신체 특성 등

## 지침별 실무 체크 (요약)

### 2.1 이미지·미디어

- 비장식 이미지: **탐지 가능**, **동등 텍스트**; 장식은 **프로그램적으로 숨김** (`alt=""` 등).
- 동영상·오디오: **대본**, **동등한 미디어 대체**, **찾기 쉬운 링크**; 자막·오디오 설명·필요 시 확장 설명.
- 정보 전달에 **색·깊이·소리·공간 오디오만** 쓰지 않기.

### 2.2 텍스트·문장

- 본문 **최소 200% 확대**, 전경/배경 **사용자 조정** 시 기능 유지.
- 보이는 텍스트는 **프로그램적으로 판별 가능**; `lang` 등 **자연어** 표시.
- 날짜·온도·시간·로마 숫자 등 **수치 메타데이터**로 모호함 제거.
- 약어 첫 사용 설명, (정보 목적 텍스트에서) 과도한 중첩·불필요한 난해 표현 줄이기.

### 2.3 대화형

- **키보드 포커스**가 보이고 대비·크기가 충분(커스텀 시).
- 포인터: 활성 표시, 가능하면 표준 포인터·가시성.
- **의미 있는 포커스 순서**, 스킵 링크·반복 내비는 건너뛰기.
- 컨트롤: **이름·역할·값·상태** 노출, **레이블과 프로그램적 이름** 정합.

### 2.4 입력·조작

- **키보드만**으로 동일 기능; 키보드 함정 없음, 포커스 이동은 사용자 통제(예외는 명세 따름).
- 복합 포인터(드래그·핀치 등)만으로 막지 말고 **단순 대안** 제공.
- 호버/포커스로 뜨는 레이어: **닫기**, 포인터 이동 시 유지 등 WCAG 2.1.13 계열 습관 유지.

### 2.5 오류

- 오류는 **식별·텍스트 설명**; 가능하면 **수정 제안**, 필드와 **프로그램·시각적 연계**.
- 여러 감각 채널(색+텍스트+아이콘 등)로 표시.

### 2.6 애니메이션·움직임

- **깜빡임·장시간 모션** 최소화; 트리거 전 **경고**와 대안 경로.
- 햅틱·오디오 패닝 등은 **끄거나 줄이기**.

### 2.7 레이아웃·구조

- **페이지/뷰 제목**, 다단계면 **단계 목록·현재 단계** (`aria-current` 등).
- 랜드마크·제목 계층·목록 마크업으로 **관계** 전달.
- 모달·오버레이는 **닫기** 가능.

### 2.8~2.12 일관성·프로세스·도움말·사용자 제어

- 반복 내비: **순서·레이블 일관**.
- 시간 제한: **연장·끄기** (본질적 제한 제외), **사전 고지**.
- CAPTCHA 등 **인지 과제**에는 대체 수단.
- 자동 재생 오디오: **조절/중지**; 자막·대본 **검색 가능** 권장.
- 동적 콘텐츠·포커스 이동: **프로그램적 알림** (라이브 리전 등).

## 접근성 지원 (Accessibility supported)

쓰는 HTML/ARIA/CSS 패턴은 **실제 브라우저·보조기술 조합에서 동작**해야 합니다. 팀은 **접근성 지원 세트**(어떤 브라우저+스크린 리더로 검증하는지)를 문서화하는 것이 좋습니다.

## 작업 순서 제안

1. **범위**: 어떤 페이지/뷰·프로세스(결제, 가입 등)를 맞출지 정의.
2. **자동**: eslint-plugin-jsx-a11y, axe, Lighthouse 접근성 등.
3. **수동**: 키보드만 시나리오, 스크린 리더 샘플, 200% 확대·고대비.
4. **기록**: 선언이 필요한 항목(스타일 가이드 검토, 사용성 테스트)은 담당·날짜·범위를 남김.
